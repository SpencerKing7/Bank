## What to do next
1. **Fix `joinGame` (highest value â€” this is the "people joining" lag).** It is still `runTransaction` with **two reads + a commit** (~3 network legs; a roll is now 1). The `tx.get(playerRef)` existence check is redundant â€” `firestore.rules` already enforces create-only semantics for player docs:
   - `allow create: if request.auth.uid == playerId && request.resource.data.points == 0` â€” only succeeds when the doc does not exist
   - `allow update: ... .hasOnly(['points','bankedRound'])` â€” a rejoining player's `setDoc` carries `name`/`joinedAt`, so it is **denied**, which preserves their points
   So: catch `permission-denied` from `setDoc` and treat it as "already joined" (the current transaction's silent `return`). **The one thing that genuinely needs a read is the `GameNotFoundError`** â€” Firestore will happily write `games/XXXX/players/{uid}` even when `games/XXXX` does not exist, so a bad code would silently "work". Keep one `getDoc(gameRef(code))` for that. Net: 2 reads + commit â†’ 1 read + 1 latency-compensated write.
2. **Fix `createGame`.** Also `runTransaction`, in a loop of up to 5 attempts, for the code-collision check. Same trick may apply: with no `allow update` path for a non-host, a `setDoc` onto someone else's existing game is denied â†’ that denial *is* the collision signal. **Caveat that needs thought:** colliding with a game *you already host* would pass the update rule and clobber it. Note it also writes the host's player doc in the same transaction â€” that part is a genuine two-doc atomic write, so consider `writeBatch` rather than dropping atomicity.
3. **Consider moving off `nam5`.** Worth tens of ms per write, not seconds â€” but the DB is ~1 day old and near-empty, so it is the cheapest it will ever be. Location **cannot be changed** on an existing database; requires creating a new one (Firestore supports multiple named DBs per project, or delete+recreate `(default)`).
4. **Optional: optimistic BANK button.** `bank()` must stay a transaction (correctness), so the player waits ~240ms+ staring at an unresponsive button. An optimistic pending state that rolls back on `BankTooLateError` would make it feel instant without touching the race.

## Attempted approaches â€” what worked and what didn't
- **Browser-pane E2E verification â€” repeatedly FAILED, do NOT sink time into it again this way.** Root cause chain: the Firebase emulator restarted and wiped its auth users, but the app restores a *cached* token from IndexedDB with **no network call**, so `onAuthStateChanged` fires with a user the emulator no longer knows and `signInAnonymously` never runs (emulator reports 0 accounts, app makes zero network requests, nothing works). Then, trying to clear it, `indexedDB.deleteDatabase('firebaseLocalStorageDb')` was called **while the SDK held the connection open** â€” that request is **origin-scoped and blocks every subsequent `open` on `localhost:3001`**, including from brand-new tabs. **Do NOT retry `deleteDatabase` while the app is loaded.** A fresh browser session should clear it; otherwise clear site data for `localhost:3001` out-of-band before relying on the pane.
- **Component tests with `@testing-library/react` â€” WORKED, use this instead.** `Game` is purely presentational (takes `game`/`players` as props), so it renders directly. Two required tricks, both non-obvious (see Gotchas): mock `EndGameButton`, and scope queries with `within(container)`.
- **Bundle-hash comparison to check what Vercel deployed â€” USELESS, don't repeat.** Local build produced `main.81bc8098.js`, deployed was `main.adc3aeb5.js`; a control build of `HEAD~1` gave `main.fd6b69db.js` â€” matching *neither*, proving Vercel's builder differs from this machine and hashes carry no signal. Bundle **sizes** are also not comparable (CRA prints gzipped, `curl | wc -c` is raw). **What worked:** `curl` the deployed bundle and `grep -o '.\{240\}<distinctive-literal>.\{60\}'` to read the minified code directly (used `status:"active",lastAction:{type:"roundStart"}` to confirm `startGame` takes 2 args and has no `tx.get`).
- **Backgrounding / iOS-eviction theory â€” DISPROVED, do not re-chase.** Simulated a 30s phone sleep with `disableNetwork`/`enableNetwork` against prod: the screen is stale only while the device is out (expected â€” nobody is looking), then **caught up in 275ms**, `onSnapshot`'s error callback never fired, and `useGame`'s 1sâ†’15s backoff never engaged. ## Active decisions / constraints
- **`bank()` must stay a `runTransaction`.** A player, unlike the host, cannot trust its own copy of the game because the host is writing it concurrently. Reading the game inside the transaction makes its version a commit precondition â€” that is what makes bank-vs-bust-7 resolve one way only. Verified to still hold against plain host writes.
- **The game doc has exactly one writer (the host's client), enforced by `firestore.rules`.** This is the entire justification for the plain-write refactor. If a rule ever lets anyone else write `games/{code}`, the host write path must go back to transactions.
- **`advanceRoundNow` has no `roundNum` precondition on purpose.** The caller's effect can fire twice, but the second pass reads a `game` already carrying the local echo of the first write (new `roundNum`, `rollNum` back to 1), so `allPlayersBanked` is false. The `advancing` ref guards the synchronous double-fire.
- **`endGame` was deliberately left alone.** Its `getDocs` is a fresh server read; using the host's cached `players` risks missing a just-joined player and orphaning their doc into a future game reusing the code. It is rare and dialog-gated â€” correctness > ~200ms.
- **The layout must fit the viewport with no page scroll** (`fullHeight`, `clamp`/`dvh` sizing). Any new list scrolls inside its own `overflow-y: auto` box â€” the new standings follow this.
- **`includeMetadataChanges: true` is load-bearing** â€” `fromCache` drives the "Reconnectingâ€¦" indicator and gates the round-advance effect. Don't remove it to reduce renders.

## Gotchas / known issues
- **`react-router-dom` v7 is ESM and CRA's Jest cannot resolve it.** Any test importing `Game.tsx` transitively hits it via `EndGameButton`. Mock it: `jest.mock('../components/EndGameButton', () => ({ __esModule: true, default: () => null }))`.
- **MUI `SwipeableDrawer` keeps children mounted and portals to `document.body`.** So `screen.getByText(...)` finds the *drawer's* `StandingsList` even when closed, and finds duplicates when the inline one renders. Scope with `within(container)` â€” portals fall outside `container`.
- **`get_page_text` reads `innerText`, which never includes `<input>` values.** An input showing its label/placeholder in that output does **not** mean the field is empty â€” check `input.value` via JS. This caused a long false trail.
- **Browser-pane clicks are in screenshot space (800x450) while the CSS viewport may be emulated at 1280x720** â€” a 0.625 scale mismatch that silently lands clicks on the wrong element. `resize_window` to 800x450 makes it 1:1. Also `left_click` with `coordinate` requires a prior `screenshot`.
- **`firebase-tools` must be pinned to `@13`** â€” this machine has Java 17 and `>=14` requires Java 21. Start: `npx -y firebase-tools@13 emulators:start --project demo-bank --only auth,firestore`.
- **The emulator drops `firebase-debug.log` / `firestore-debug.log` in the repo root** and they are **not** gitignored â€” delete before committing.
- **Purge emulator state between runs** or stale docs owned by a previous run's anonymous uid will trip the rules: `curl -X DELETE -H "Authorization: Bearer owner" "http://localhost:8080/emulator/v1/projects/demo-bank/databases/(default)/documents"`. Inspect with the same `Bearer owner` token; it also `PATCH`es state the rules forbid a client from creating.
- **Emulator green â‰  production green.** Rules changes only take effect in prod after publishing (console paste, or `firebase deploy --only firestore:rules`).
- **The reported tsconfig error was not real** â€” the config typechecks clean under both TS 4.9.5 and 5.9; it was a stale editor TS server and the user confirmed it resolved itself.
- **`public/manifest.json` is still the untouched CRA default** ("React App" / "Create React App Sample"). Unrelated to latency, but wrong for a PWA-ish installed game.
- **The app has no Wake Lock and no `visibilitychange` handling.** Not currently implicated (backgrounding was disproved), but a party game where players watch and wait will let phones dim/lock mid-round.
